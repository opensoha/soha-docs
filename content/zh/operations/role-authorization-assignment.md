# 角色授权分配运行手册

## 目的

本手册用于标准化 soha 中的角色授权分配流程，避免“角色已经分配，但页面仍不可见”或“页面可见，但接口仍然 403”这类常见偏差。

当前控制台权限已经收敛到“`permissionKeys` 派生菜单可见性”为默认路径：

- 角色上的 `permissionKeys` 决定页面访问、后端控制面 API 权限，以及大多数菜单可见性
- 菜单与角色的显式绑定只保留给少数例外或未映射菜单

对平台类资源页面，还可能叠加第三层约束：

- ABAC 策略或 `scope grants` 决定的集群、命名空间与资源范围

这意味着“给用户分配一个角色”本身仍不等于“用户已经拥有完整可用的页面与 API 访问能力”，但常规页面已不再需要额外手工维护一份菜单角色绑定。

## 适用范围

本手册适用于以下运维或管理员场景：

- 新增角色并授予控制台页面访问权限
- 给现有角色增加读写能力
- 排查某个用户为什么看不到菜单、打不开页面、不能保存配置、或看得到页面但拿不到平台资源数据

## 生效模型

角色授权是否完整生效，按下面顺序判断：

1. 用户是否绑定了目标角色
2. 目标角色是否包含所需 `permissionKeys`
3. 对应菜单是否属于默认派生路径，还是仍需显式绑定的例外路径
4. 如果是平台资源或范围化数据，用户或其角色/组织是否还有对应的策略与 `scope grants`
5. 用户当前会话是否已经刷新到最新的权限快照

只要其中任何一层缺失，最终体验都可能是部分可用或完全不可用。

## 前置准备

执行授权前，先确认以下信息：

- 目标用户是谁
- 目标用户属于哪些组织
- 目标角色是新建角色还是已有角色
- 要开放的是页面可见、只读、还是可写/可操作能力
- 涉及哪些菜单
- 涉及哪些 `permissionKeys`
- 目标菜单当前在菜单管理里显示为 `自动派生`、`显式覆盖` 还是 `未映射`
- 是否还需要平台资源范围授权，例如集群、命名空间、标签选择器

建议在变更单或操作记录中提前写清下面三类映射：

- 页面或功能 -> 对应菜单
- 页面或功能 -> 对应 `permissionKeys`
- 页面或功能 -> 是否需要 `scope grants` 或 ABAC 策略
- 页面或功能 -> 是否走默认派生，或需要显式菜单例外

## 正式分配检查清单

### 1. 明确目标能力

先把需求翻译成系统内可执行的授权对象，而不是只写“给某某开放权限”。

至少要明确：

- 目标页面或模块，例如 `访问控制 > 用户`、`设置中心 > AI`
- 是只读还是可管理
- 是否需要后端写接口
- 是否需要平台资源范围

推荐使用下面的拆分方式：

- 页面访问：`*.view`
- 变更或保存：`*.manage`
- 平台运行时操作：由资源 `capabilities`、`allowedActions`、策略与范围控制

### 2. 校验或更新角色的 `permissionKeys`

在角色定义中确认目标角色已经具备所需 `permissionKeys`。

典型示例：

- 访问用户列表：`access.users.view`
- 创建或更新用户：`access.users.manage`
- 查看 AI 设置页：`settings.ai.view`
- 修改 AI 设置：`settings.ai.manage`

检查点：

- 只读场景不要误配成仅有 `*.manage` 而缺少 `*.view`
- 可写场景通常需要同时具备 `*.view` 和 `*.manage`
- 不要把平台资源动作能力误写进 `permissionKeys`
- 不要假设内置角色默认值一定等于当前数据库中的持久化角色

### 3. 校验菜单可见性模式

大多数已接入的控制台菜单不再要求手工做菜单角色绑定。先在 `系统管理 > 菜单管理` 确认当前菜单属于哪一类：

- `自动派生`
  - 这是默认路径
  - 角色具备对应 `permissionKeys` 后，菜单会自动进入 `visibleMenuIds`
  - 不需要再手工维护 `roleIds`
- `显式覆盖`
  - 只用于少数例外场景，或运维明确希望保留单独菜单绑定时
  - 当前实现里，`roleIds` 是补充可见性的回退路径，不是“比权限键更强的隐藏规则”
- `未映射`
  - 说明前端路由和当前后端派生表还没有形成稳定映射
  - 这类菜单如果仍要控制可见性，应填写显式 `roleIds`

判断原则：

- `permissionKeys` 解决“是否允许访问页面/API”
- 菜单派生或显式绑定解决“该入口是否进入可见菜单集合”

对已映射菜单，常规流程只维护 `permissionKeys`；只有例外路径才维护 `roleIds`。

### 4. 校验平台范围授权

如果页面属于平台资源或受范围控制的数据面，菜单可见和页面可打开仍然不代表用户一定能看到资源数据。

需要继续确认：

- 用户角色或组织是否命中了正确策略
- 是否存在对应 `scope grants`
- 集群、命名空间、标签条件是否覆盖目标资源

典型症状：

- 页面能打开，但列表为空
- 详情页能进入，但操作按钮消失
- 后端返回允许登录，却对具体资源请求返回 403 或空结果

### 5. 把角色分配给用户

完成角色定义后，再确认用户本身已经绑定目标角色，并且组织关系正确。

检查点：

- 用户角色绑定是否保存成功
- 组织绑定是否同步保存成功
- 如果策略依赖组织，不要只绑角色不绑组织

如果用户通过 OIDC、OAuth2、飞书、钉钉或企业微信登录，并且登录源开启了“登录补充角色”或“登录补充组织”，还要确认：

- `roleField` 和 `organizationField` 与 provider 返回的 claim/profile 字段一致
- 外部角色引用能匹配到本地 `roles.id` 或 `roles.name`
- 外部组织引用能匹配到本地组织的 `id`、`slug`、`org_path`，或匹配到组织的 `source + externalId`
- `replace_external` 只会替换同一登录源写入的绑定，不会删除用户页面手工维护的本地绑定

### 6. 刷新权限快照

角色、菜单、策略、范围变更后，当前登录态未必立即拿到最新权限快照。

操作后应执行至少一种刷新动作：

- 重新登录
- 刷新当前会话对应的认证上下文
- 在浏览器里确认最新会话请求已经返回更新后的 `permissionKeys` 与 `visibleMenuIds`

如果这一步跳过，最常见结果是后台数据已经改对，但前端仍表现为旧权限。

## 验证步骤

### 基础验证

1. 使用目标用户重新登录控制台
2. 检查目标菜单是否出现在侧边栏
3. 打开目标页面，确认没有被路由拦截或跳转
4. 对只读页面，确认列表或详情可正常加载
5. 对可写页面，确认保存、创建、删除或其他管理按钮按预期显示

### 权限快照验证

在浏览器开发者工具的网络请求中，检查登录后会话初始化或权限快照返回的数据，确认：

- `permissionKeys` 包含目标键
- `visibleMenuIds` 包含目标菜单

如果目标菜单属于 `自动派生`，通常应同时命中这两项；如果属于 `显式覆盖` 或 `未映射`，则还要继续核对菜单配置本身。

### API 验证

对目标功能实际发起一次读请求和一次写请求。

验证目标：

- 只读角色：读请求成功，写请求被拒绝
- 管理角色：读写请求都成功
- 平台范围角色：请求只返回被允许的集群、命名空间或资源

### 范围验证

如果涉及平台资源，再额外验证：

- 切换到允许的集群/命名空间时可见数据
- 切换到未授权范围时为空或被拒绝
- 行级操作只在允许范围内出现

## 常见失败模式

### 角色已有 `permissionKeys`，但菜单仍不可见

现象：

- 侧边栏看不到页面
- 页面可能被路由拦截，或入口不可发现

处理：

- 先确认该菜单是否是 `自动派生`
- 如果是 `自动派生`，检查菜单是否 `enabled`、菜单 ID 是否已接入后端派生表、以及会话是否已刷新
- 如果是 `未映射`，改用显式 `roleIds` 或补齐前后端映射后再验证

### 菜单已显式绑定，但 `permissionKeys` 缺失

现象：

- 菜单可能可见，但页面打开后被权限校验拦截
- 页面能进，但保存操作返回 403

处理：

- 给角色补齐对应 `*.view` 或 `*.manage`

### 已配置显式 `roleIds`，但有权限的用户仍然能看到菜单

现象：

- 菜单处于 `显式覆盖`
- 某用户不在显式 `roleIds` 里，但只要具备对应派生权限键，菜单仍可见

处理：

- 这是当前实现行为，不是单独故障
- 当前后端将显式 `roleIds` 作为补充可见性的回退路径，而不是收窄派生权限结果的强制覆盖
- 如果需要真正的“有权限但仍隐藏”语义，需要后续产品与后端模型专门收敛

### 未映射菜单未配 `roleIds`，结果默认可见

现象：

- 菜单管理里显示 `未映射`
- 菜单启用且没有显式 `roleIds`
- 用户即使没有特定页面权限，也能在侧边栏看到该入口

处理：

- 这是当前兼容路径，用于避免旧菜单因尚未接入映射而突然消失
- 若该入口需要权限控制，补齐显式 `roleIds`
- 若该入口应收敛为默认派生路径，则安排补齐后端菜单 ID 到权限键映射

### `view` 和 `manage` 只配了一半

现象：

- 页面可见，但保存按钮缺失
- 页面按钮可见，但提交时被后端拒绝

处理：

- 对照页面语义分别补齐 `*.view` 与 `*.manage`

### 角色已更新，但用户会话还是旧快照

现象：

- 数据库或后台配置已经正确
- 前端仍按旧权限显示

处理：

- 重新登录或刷新权限快照

### 页面可见，但平台数据为空

现象：

- 菜单和路由都正常
- 资源列表为空，或资源操作被拒绝

处理：

- 检查 ABAC 策略与 `scope grants`
- 检查组织、角色、命名空间和集群范围是否真正命中

### 误把运行时资源动作当成控制台页面权限

现象：

- 给了 `permissionKeys`，但重启、扩缩容、日志、终端等动作仍不可执行

处理：

- 检查资源 `capabilities`
- 检查后端返回的 `allowedActions`
- 检查对应范围策略

## 变更后最小验收标准

一次正式授权变更至少应满足以下结果：

- 目标用户能看到正确的菜单入口
- 目标页面可打开
- 只读与可写能力符合预期
- 平台数据范围符合预期
- 未授权页面、按钮和范围仍保持拒绝

如果上述任一项未通过，不应把本次授权变更视为完成。

## 应用交付工作台角色矩阵

应用交付工作台使用 `workspace.application.view` 打开一级工作台，再按 `delivery.*` 权限键控制菜单、页面和按钮。常规角色口径如下：

| 角色 | 可见入口 | 允许动作 | 禁止动作 |
| --- | --- | --- | --- |
| `admin` | 应用中心、应用接入、构建发布、测试验证、问题分析、版本包、执行任务、流程模板、构建模板、环境绑定、镜像仓库 | 完整管理应用交付配置、模板、绑定、仓库和触发交付动作 | 无默认交付限制，仍受审批、审计和范围策略约束 |
| `ops` | 应用中心、构建发布、模板、环境绑定、镜像仓库和执行记录 | 管理交付平台配置、环境绑定、模板、仓库，触发构建、工作流和发布 | 不应绕过审批或范围策略直接操作未授权环境 |
| `developer` | 应用中心、应用接入、构建发布、测试验证、问题分析、版本包和执行任务 | 维护应用档案和服务组件，发起非生产交付动作，查看交付证据 | 不默认管理全局模板、镜像仓库或生产范围外资源 |
| `tester` | 应用中心、测试验证、问题分析、版本包、执行任务和证据类内容 | 查看应用、候选版本、执行任务和验证证据 | 不能进入构建发布、应用接入、模板、环境绑定或镜像仓库维护入口；不能触发构建、发布、回滚或保存配置 |
| `readonly` | 应用中心、测试验证、问题分析、版本包、执行任务、工作流和发布记录 | 只读查看交付对象、运行记录和证据 | 不能触发构建、发布、回滚、工作流；不能保存模板、环境绑定、镜像仓库或应用配置 |

按钮级验收应至少覆盖这些边界：

- `readonly` 打开应用详情时，`构建`、`部署`、`构建并部署`、`运行验证` 必须禁用且不会调用交付动作 API。
- `readonly` 打开构建模板、发布流程模板、环境绑定、镜像仓库页面时，新建、保存、复制、删除、编辑和启用开关必须隐藏或禁用。
- `tester` 的菜单和路由只包含应用、测试验证、问题分析、版本包、执行任务和证据类入口。
- 新增交付页面、按钮或菜单时，必须同时补 `permissionKey`、菜单 seed、路由元数据和页面级按钮测试。

## 操作建议

- 默认先维护 `permissionKeys`，不要把显式菜单绑定当成常规步骤
- 在菜单管理里优先让路由型菜单保持 `自动派生`
- 新增页面或新权限时，同步维护“权限键 + 菜单映射/例外模式 + 范围要求”
- 新增自定义菜单时，明确它是：
  - 走后端派生表的正式菜单
  - 暂时通过显式 `roleIds` 控制的例外菜单
  - 尚未收敛、因此保留兼容可见性的未映射菜单
- 不要只验证页面可见，还要验证真实 API 与真实范围
- 自定义角色上线前，先用一个测试用户完整走一遍登录、菜单、页面、写操作、范围过滤验收

## 未决例外路径

- 已映射菜单上的显式 `roleIds` 当前只提供补充可见性，不会压过派生权限结果
- 未映射且未配置 `roleIds` 的启用菜单，当前仍保持默认可见以兼容历史菜单
- 前端菜单管理的 `自动派生/显式覆盖/未映射` 标签，当前部分依赖路由元数据推断；后端持久层仍只保存 `roleIds`
